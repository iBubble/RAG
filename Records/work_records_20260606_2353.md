## 2. 修改细节说明

### A. 全局触发机制的优化（worker.py）
- 之前的触发机制限制于单文件 `total_triples > 0` 且全局 `slow_queue` 长度清空。这导致当最后一个处理完的文件本身不产生三元组时，整个项目的图谱后处理与摘要任务会被永久丢失。
- 优化后，改为在每个文件图谱提取结束时，**自动扫描当前项目下所有上传文件的状态**。
- 仅当检测到项目下的所有文件均已退出图谱排队或提取阶段（状态非 `"graph_queued"`、`"graph_extracting"` 等）时，正式判定项目图谱已全量结束：
  - 如果图谱检测到的实体总数 `total_entities > 0`，自动平滑触发 Phase 1 实体消歧和 Phase 2 社区发现任务。
  - 如果图谱内没有任何实体（`total_entities == 0`，如某些空白测试项目），直接将 Redis 里的社区摘要进度设置为已完成（`completed`），避免出现挂起灰字。

### B. 自动补偿与状态修正兜底（api/admin.py）
- 在 `/api/admin/learning-progress` 接口中，针对已完成图谱提取的项目加入自愈检测：
  - 如果图谱提取已 100% 结束且实体总数为 0，接口在获取状态时自动将社区摘要进度置为 `completed` 与 `100`%，使前台直接高亮显示为“全局知识摘要完毕”。
  - 如果图谱提取已 100% 结束，实体数 > 0，但社区摘要依然因为历史 Bug 等原因遗留在 `pending` 且总数为 0 状态时，**后端主动进行任务补发补偿**，自动投递 Celery `compute_community_summaries` 任务进行后台自愈计算。

### C. 平滑服务载入
- 执行了 PM2 平滑重启命令 `pm2 reload all`，后端服务在未出现任何连入中断的情况下平滑应用了最新逻辑。根据系统日志，原先挂起处于 `pending` 的各已有知识库项目已被成功捕获，并在重启后的第一次状态获取时全自动补偿执行了社区摘要。
