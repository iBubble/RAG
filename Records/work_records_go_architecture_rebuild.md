## 🛠️ 改造背景与核心痛点
原系统基于纯 Python 单体/异步架构（FastAPI + Celery + Uvicorn）。在高频多用户协同聊天、流式 SSE 传输以及批量 RAG 分析等重负载高并发场景下，遇到以下瓶颈：
1. **Python GIL 限制**：CPU 密集的文本解析或格式化阻碍了 I/O 调度，打字机流吐字卡顿、首字延迟波动。
2. **连接稳定性差**：缺乏有效的长连接保活与级联取消机制，大模型耗时长思考时，SSE 易被中间反向代理掐断报错，或前端关闭时后台大模型仍在空转。
3. **并发连接开销高**：长连接挂载和频繁的心跳上报消耗了大量 Python 后端服务资源。

## 📝 改造内容与核心架构设计
我们对系统实施了彻底的**混合并发微服务化改造**：

1. **引入 Go 核心网关 (Nexus-Gateway)**：
   - 使用 Go 语言和 Gin 框架编写高性能外网流量网关，承载 JWT 校验、高频长连接和状态 indicator 维护，将并发网络消耗完全屏蔽在 Go 侧。
   - 核心流式对话（`/api/chat`）引入 Go 原生 Eino 图编排框架进行重构，实现基于 Eino Graph 的流式对话流控、L2 缓存管理及中断机制。
2. **反向代理与 Python 重型算法分工**：
   - Go 网关通过 `NoRoute` 泛路由反向代理，将低频、重型的法律事务多阶段串联流（`/api/legal/workflow/stream`）及合同物理批注审核转发给 Python 算法后端，继续保持 Python 强大的生态与业务开发效率。
3. **流式保活与防断连机制 (Heartbeat Ping)**：
   - 针对非流式的耗时计算，在 Python 后端流生成器中引入了每 4 秒向客户端发送零宽字符 `"\u200b"` 的异步保活机制，彻底根治了 `network error` 代理掐断 Bug。
4. **上下文流过滤与防污染 (Context Filter)**：
   - 前端部署了定稿标志过滤器 `cleanStageOutput`，利用仲裁官（👑）标志智能剥离协同阶段的过程冗余内容，仅将最纯净的定稿结论传递给后续大模型步骤。
