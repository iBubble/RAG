## 架构设计

### 协作模式：辩论 + 仲裁

```
用户请求 → Supervisor(路由) → Worker Agent(执行) → 小杠(质疑) → 大BOSS(仲裁) → 用户
```

- **Tool Calling**: Ollama 原生 `/api/chat` tools 参数（非 /api/generate）

## 关键设计决策

### 1. 为什么用 /api/chat 而不是 /api/generate
- `/api/generate` 不支持 `tools` 参数，无法实现 Tool Calling
- `/api/chat` 原生支持 OpenAI 兼容的 tools JSON Schema
- Qwen3 系列模型在 Ollama 中已完整支持 Tool Calling

### 2. 为什么自研框架而非用 LangGraph
- LangGraph 默认依赖 OpenAI API，与 Ollama 原生接口需适配层
- 自研可完全复用现有 Redis/Celery/GPU 调度器
- 减少 Docker 镜像体积和部署复杂度

### 3. 简单问题的快速路径
- Supervisor 判断为简单问候时 → direct_answer，跳过小杠/大BOSS
- Worker 回答少于 100 字时 → 跳过辩论环节
- 保证简单问题不会因多 Agent 链路而变慢
