# 经验总结：Markdown 渲染中引用标记被误识别为 H1 标题的格式治理

在 RAG 系统的流式及静态 Markdown 渲染场景中，经常会出现“流式输出时排版正常，但生成完毕后重新渲染时突然发生异常换行、且断开后的部分字体莫名其妙变大、变粗”的奇特 Bug。

本文总结此 Bug 的定位过程、技术根源以及标准防范治理经验。

---

## 一、现象描述
1. **流式传输阶段 (Streaming)**：
   大模型正在打字输出，此时内容显示完全正常，没有任何断行或字体粗细问题。
2. **流式结束阶段 (Finished)**：
   大模型停止生成，前端将临时状态切换为最终的 Markdown 渲染。此时排版瞬间发生剧烈形变：
   * 完美的 `根据**【参考文档区块 #1】...` 句子在中括号内部突然断开。
   * 断开后的 `1】《案例一...` 独立成为新的一行，且字号异常粗大、呈现为 H1~H2 标题级别的黑体加粗。

---

## 二、定位与根源分析

### 1. 表面元凶：Vite 打包乱码
起初曾怀疑是由于大模型在生成中文引用时夹杂了 `\n`，但在前端加上针对中文 `【参考文档区块` 的正则替换后发现依旧无效。这是因为 Vite 或打包工具在进行生产编译（Minify）时，如果正则中包含中文字符字面量，其编码会被转译或污染，导致浏览器运行时的 JS 正则与实际中文内容发生“静默失配”，正则替换根本没有执行。

### 2. 终极真凶：过度防御的 Markdown 自动修正正则
在 `normalizeMarkdown` 模块中，有一行遗留的防御性正则：
```typescript
text.replace(/([^\n])(#{1,6})/g, '$1\n\n$2')
```
* **原始设计初衷**：为了让大模型在漏掉标题换行时，能自动在标题符号（`#`）前补全 `\n\n` 以防 Markdown 解析失败。
* **灾难连带反应**：
  大模型输出的原始字符串实际上是完美无缺的：`根据**【参考文档区块 #1】...`。
  但上述正则在匹配到 `#1` 时，由于 `#` 前面是 `区块 `（一个非换行符），导致它在 `区块 ` 与 `#1` 之间**强行插入了两个换行符 `\n\n`**。
  在 Markdown 标准规范中，**任何新的一行如果是以 `#` 开头，它都会被无条件解析为一级标题（H1）**！
  所以，`#1】...` 突然在新的一行以 `#` 开头，被 Marked 解析为大字号的 `<h1>` 标题，造成了断行加字号变大的双重格式 Bug。

---

## 三、经验与最佳实践

### 1. 通用括号边界清洗正则（防打包乱码）
不要在前端正则中写死具体的中文业务词汇。应直接基于语法边界提取并压缩，采用 `Unicode` 编码防止打包转译：
```typescript
// 1. 中文中括号 【...】 内部所有换行及空格强行压缩回一行
let cleaned = text.replace(/【([^】]+)】/g, (_, p1) => {
  return '【' + p1.replace(/[\r\n]+/g, '').replace(/\s+/g, '') + '】';
});
// 2. 英文中括号 [...] 内部所有换行及空格强行压缩（排除 Markdown 链接格式）
cleaned = cleaned.replace(/\[([^\]\(\)]+)\]/g, (_, p1) => {
  return '[' + p1.replace(/[\r\n]+/g, '').replace(/\s+/g, '') + ']';
});
```

### 2. 严格限制 `#` 强插换行正则的范围
绝对禁止在全局简单匹配 `(#{1,6})`。在正文中出现的 `#1` 序号、`#FFF` 颜色值均会产生误伤。如需修正标题格式，应严格要求 `#` 号后必须跟有空格，且只能处理前置带有单换行符的情况：
```typescript
// 仅允许给符合规范 of 行首标题（有空格）且前置单换行的符号补全双换行
text.replace(/([^\n])\n(#{1,6}\s)/g, '$1\n\n$2')
```

通过本案例，开发人员应警惕任何“旨在美化 Markdown 格式的过度正则处理”，并注意本地开发环境与打包生产环境的编码一致性。
