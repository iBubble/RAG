## 1. 市场监督局业务场景特征与类似政务法律RAG项目深度剖析

市场监督局作为国家行政执法与市场秩序维护的核心职能部门，其日常业务涉及企业准入登记、行政处罚、食品药品安全监管、知识产权保护、消费者权益保护以及反不正当竞争等多元化领域。在这些业务场景中，公文编写、行政处罚决定书定制以及各类行政审批表格的自动填写具有极高的高度严肃性与法律合规性要求。任何非结构化法律条文的引用偏误、企业实体的关键元数据缺失或计算公式错误都可能导致行政复议或诉讼失败，因此该业务场景对基于检索增强生成（RAG）系统的输出精度、事实一致性与可追溯性提出了极近苛刻的要求。

在国内外公共管理与政务数字化转型进程中，已涌现出一批具有代表性的RAG系统与研究范式：

*   **KemenkeuGPT**（印尼财政部）：将RAG技术引入政府内部行政管理，基于循证决策的概念，为政策制定、合规审查和跨部门行政决策提供权威的知识辅助，验证了RAG系统在保障政务数据主权、完全私有化部署以及规避数据泄露风险方面的可行性与安全价值。
*   **ChatLaw 和 Lawyer LLaMA**（中文法律与监管RAG）：展示了特定领域微调模型与RAG系统相结合的演进路径。虽然对大模型进行法律特定数据的指令微调（Fine-tuning）能够使其回答风格和术语更契合行政标准，但微调成本通常是RAG框架的五倍以上，且微调模型极易产生更严重的“幻觉”现象。相比之下，ChatLaw 通过引入多路检索融合、外部法条库动态关联等RAG手段，显著缓解了幻觉，并实现了法条引用的精准追溯。
*   **RAGFlow (DeepDoc) 与 ChatDOC (pdflux)**：在底层文档解析与结构化检索的技术方案中，RAGFlow 强调在文本向量化之前必须精准识别文档物理布局，其内置的 DeepDoc 视觉理解引擎融合了 OCR（光学字符识别）、DLR（文档布局识别）和 TSR（表格结构识别）三大模块，不仅能过滤页眉页脚，还能精准还原含有跨行跨列（Rowspan/Colspan）的复杂表格。而 ChatDOC 平台的 pdflux 解析器在数千万页真实文档训练基础上，提出基于物理文档对象检测的结构化分块策略，实际测试中的检索表现明显优于依赖 PyPDF 和递归字符分割器（RecursiveCharacterSplitter）的传统 Naive RAG 系统。

## 2. 现有 AgentRAG 架构评估与功能改进方向

当前正在开发和调试的 AgentRAG 系统是一个高度本地化、安全且专为企业级公文及填表场景打造的私有化 RAG 协作平台。其底座架构设计合理，通过 Go 语言编写的高性能 Nexus-Gateway 解决高并发连接，并采用 Eino 图编排框架管理大模型对话流；Python 算法后端利用 FastAPI 负责复杂的 ASR、Tesseract OCR 图像文本提取以及 OpenXML 级别的 Microsoft Word 强制留痕修订生成。在计算资源受限的本地硬件环境（如 Mac Studio 统一内存架构）下，系统通过 PM2 进程守护、Celery 单并发慢任务队列以及“Vector Only”（仅向量检索）等动态算力控制模式，物理性地避免了内存溢出（OOM）问题，具有极高的私有化工程实用价值。

然而，针对市场监督局对公文编写法律准确性、复杂表格自动填写的确定性以及行政审批全链路审计追溯的极高标准，AgentRAG 在现有技术路径上依然存在三个亟待突破的核心短板：

1.  **缺少视觉版面感知与表格原子结构保护能力**：尽管 AgentRAG 目前设计了三级回退分块算法（段落 → 章节大纲 → 标点）并对 Markdown 表格进行了原子级整行保护，但在面对市场监督局经常收到的扫描件 PDF、双栏排版行政通告、印章遮挡的申报表以及边框缺失的资产负债表时，基于纯文本规则的分块算法极易割裂上下文。Tesseract OCR 引擎在处理低质量扫描件和复杂版面时，其文本流极易发生左右栏混淆、表格列数据错位以及公式断裂，导致 Qdrant 向量库中存储的 Chunks 充斥着空间逻辑错位的噪音，在检索源头便污染了 LLM 的上下文。
2.  **缺乏确定性结构化填表的约束解码机制**：当前系统在自动填写登记表或生成结构化 JSON 数据时，依然面临大模型生成格式崩溃的潜在风险。本地运行的 Qwen-35B 尽管能力强大，但在未施加硬性约束的前提下，难以保证生成的每个 JSON 数据都严格符合市场监督局内部审批系统（AcroForm/PDF/Excel）的 Schema 规范。一旦模型在生成的 JSON 中添加了 Markdown 的 ```json 代码包裹框，或者在必填字段处因幻觉生成了无法解析的自然语言，下游的 .NET OpenXML 渲染引擎便会报错崩溃，无法完成完全自动化的审批填表闭环。
3.  **缺乏多角色 Agent 协同及全链路合规审计留痕机制**：市场监督领域的公文写作与填表并非简单的“单轮问答”或“单向生成”。在真实业务中，一份《行政处罚决定书》从立案调查、违法事实认定、法律条款适用，到最终的处罚额度计算，需要经历定量校验与定性合规审核的双重审查，并需要法务人员进行人工复核。现有系统虽具备跨项目检索、地名纠错等功能，但缺乏“大模型 + 小模型 + 业务界面”的协同审计机制，无法留存完整的审核合规等级、风险提示和人工复核记录，难以应对外部审计与司法行政复议的追溯要求。

## 3. 核心功能演进的技术实现方法与算法路径

为彻底解决上述技术瓶颈，本系统需在原有的 AgentRAG 基础上进行功能演进与重构。其核心算法路径和具体实现方法如下：

### 3.1 引入多模态视觉版面分析与 DoCO 规范化统一节点表征
针对文档解析质量低导致的检索偏差，本方案建议将原有的 Tesseract OCR 单路解析升级为基于 **DoCO（Document Components Ontology）扩展语义图谱** 的双路智能解析管线。

*   **数字原生 PDF 与 DOCX 文件**：调用 Apache 2.0 协议的 **Docling** 引擎，将其转化为富含层级树状结构的 `DoclingDocument` 表征。
*   **扫描件与复杂表格**：针对含有复杂跨行跨列、无边框表格的市场监管历史卷宗扫描件，通过 FastAPI 异步调度 **MinerU**（利用其 97.5% mAP 的布局识别能力与 PaddleOCR-VL 视觉大模型）进行专项表格还原，并输出高保真 Markdown 表格片段。

为了屏蔽多源解析器输出格式的异构性，系统需在 Python 算法层实现解析器独立的“证据单元”（Evidence Units, EUs）构建管线：
1.  **节点规范化（Node Normalization）**：编写适配层将 Docling 输出的 `"heading"`、MinerU 输出的 `"title"` 统一映射为本体定义的标准化节点 `section_header`。
2.  **视觉种子锚定与分配（Visual Seeding & Paragraph Assignment）**：基于物理坐标的相似度算法，将图像、印章等视觉元素与最邻近的文字段落进行全局相似度关联。
3.  **图决策层持久化（Graph-Based Persistence）**：将生成的 Evidence Units 关系网同步持久化存储于 Neo4j 中，作为底层文档结构的最终权威规范。其形式化定义如下：
    $$EU_k = \{Text_k, Bbox_k, Page_k, SemanticRole_k, ParentNodeID\}$$

这种方法能够保证不论解析器如何切换，Qdrant 向量检索和 SQLite 混合检索获取的知识切片都具有绝对完整的物理、空间与语义关联性，从根本上解决表格检索割裂与信息遗漏问题。

### 3.2 基于约束解码（Constrained Decoding）的 100% 确定性表格填充
针对填表工作流中大模型输出格式极易崩溃的问题，本系统应全面引入 **约束解码** 机制，将 Pydantic 数据格式化验证前置到模型推理的每一步（Next-Token Prediction）。

在传统的自回归解码中，模型在时间步 $t$ 生成词表的概率分布，当需要生成结构化 JSON 时，容易因为采样随机性输出非 JSON 字符。而在约束解码模式下，系统将市场监督局的 Excel 或 PDF 填充模板抽象为 `Pydantic BaseModel` 模式：

```python
# app-server/backend/app/schemas/market_supervision.py
from pydantic import BaseModel, Field, field_validator
from typing import Literal

class CorporatePenaltyForm(BaseModel):
    company_name: str = Field(..., description="企业全称，需与统一社会信用代码对应")
    credit_code: str = Field(..., min_length=18, max_length=18, description="18位统一社会信用代码")
    penalty_type: Literal["警告", "罚款", "暂扣许可证", "吊销营业执照"] = Field(..., description="法定处罚种类")
    fine_amount_yuan: float = Field(default=0.0, description="罚款金额（元），无罚款则为0")
    legal_basis: str = Field(..., description="适用的具体法律法规条文主旨")

    @field_validator('credit_code')
    def validate_code(cls, v):
        if not v.isalnum():
            raise ValueError("统一社会信用代码必须由数字和英文字母组成")
        return v
```

在推理阶段，FastAPI 后端通过调用 Pydantic 导出的 JSON Schema，将其作为语法状态机（FSM）或语法约束文件传递给本地 **Ollama 0.3+** 的 `/api/generate` 或 `/api/chat` 接口中的 `format` 字段。

Ollama 的推理引擎将在每个 Token 生成步计算掩码，强制将违反 JSON 语法或字段类型的候选 Token 概率置为 0：
$$P_{mask}(t_i \mid t_{<i}) = \begin{cases} P(t_i \mid t_{<i}) & \text{if } t_i \text{ is allowed by Schema FSM} \\ 0 & \text{otherwise} \end{cases}$$

此方法的优势在实际评测中非常显著：由于模型不再耗费上下文 Token 去纠结格式标记（如 Markdown 语法围栏），**整体推理速度可提升 6.4 倍**，且输出的 **JSON 格式解析成功率达到 100%**。若发生业务逻辑字段验证失败（触发 `ValidationError`），FastAPI 后端则自动捕获错误，并将验证失败的 Trace 日志作为提示词动态反馈给 local LLM，完成毫秒级的自纠错重试闭环。

### 3.3 大小模型协同的“双路合规审计”与 Go Eino 多 Agent 异步协同图
市场监督局的文书生成和数据处理，应当遵循“大小模型协同运作、定量定性双路并行”的科学范式，全流程通过高性能 Go Gateway 实现图编排和人机交互控制。在业务执行中，系统应引入 **大模型 + 小模型 + 业务界面** 的协同运行机制：

*   **定量校验（小模型主导）**：小模型或规则库（如 Python 常用校验算法、OCR 信息抽取模块）完成对“印章完整性”、“企业注册号有效性”、“历史罚款金额累计”等边界清晰、规则硬性的定量任务进行闪电校验。
*   **定性审核（大模型主导）**：结合 RAG 检索回来的《民法典》、《消费者权益保护法》及市监局内控案例库，本地大模型（Qwen-35B）针对文书草稿中的“自由裁量权适用是否合理”、“违法事实陈述逻辑是否严密”等规则模糊的任务进行定性分析，并输出红线风险提示。
*   **多 Agent 协同与人机共创**：借助 ByteDance 的 Go 语言 **Eino** 编排框架，系统应构建包含“规划者 Agent”、“检索者 Agent”、“合规审计者 Agent”在内的多决策主体网络。由于 Go 语言具备天然的高并发优势，goroutine 在高并发 WebSocket 连接 and SSE 流式响应下能够保持极低的内存资源开销，避免了 Python 大规模并发下的性能退化。
*   **人机协同中断与复核（Interrupt & Resume）**：Eino 的图执行流程具备状态冻结机制。当“合规审计者 Agent”判定生成的处罚文书触发高风险警告（如罚款金额异常超出常规基准）时，Eino 将触发 `Interrupt`，冻结当前 Agent 节点的状态机并将其保存至 Redis 中，前端 React 界面立即弹窗提示执法人员人工介入复核。在法务人员手动修正、确认无误后，图执行流被 `Resume` 激活，继续交由 .NET 8 OpenXML 渲染生成最终的正式 PDF 或 Word 公文。

### 统一内存与物理显存（VRAM）动态调度集成方案
由于 AgentRAG 的典型部署场景是中大型企业或监管局本地的消费级 workstation（如 Mac Studio 192GB 统一内存版或双卡 NVIDIA RTX 4090），系统在集成上述组件时，必须防止重度计算组件（MinerU, Qwen-35B, Whisper-Base）同时拉起导致物理显存暴满。系统应当复用并扩充已有的物理算力控制模式：

*   **全速模式（Full Speed Mode）**：Go 核心网关将 35B 大模型常驻 VRAM。MinerU 的 OCR 与 TSR 管道配置为 CPU 线程或分配独立的 GPU 核心。当 Celery 监听到慢通道中有大型扫描件时，自适应降低 LLM 吞吐，平滑物理带宽。
*   **仅向量检索模式（Vector Only Mode）**：挂起 Neo4j 的三元组动态抽取任务与 MinerU 视觉识别，将 Qwen-35B 动态移出 VRAM 释放至多 23GB 显存，优先保证 Docling 在 CPU 端以 8.2 秒/12 页的高速对海量历史归档文书进行清洗与向量入库。
*   **挂起模式（Suspended Mode）**：PM2 强行冰冻 Python 端 Celery 异步慢队列。此时，除必要的 Redis L2 缓存及 SQLite 快速检索服务外，系统将整机算力开销和 CPU 占用率降至 0%，仅保留最基础的 Web 接口响应，等待管理员人工拉起。

## 5. 监管级质量评测、RAG三元组度量与回归基准

公文填写和合规决策的高精度要求，决定了不能将 RAG 系统的质量改进诉诸于非量化的“主观感觉”。必须为市监局知识库建立一套无偏、自动化且能够私有化闭环运行的评测回归基准。

### 5.1 RAG 三元组评估范式（RAG Triad）
本系统应集成 **Ragas** 与 **Langfuse**，以大模型作为裁判（LLM-as-a-judge）来对每一次填表、每一次公文生成调用进行闭环自动化评估。

*   **上下文相关性（Context Relevance）**：验证检索出来的《特种设备安全监察条例》等法律片段是否高度切中当前案情，剔除不相关噪音，保护上下文窗口。
*   **忠实度/接地度（Groundedness）**：计算大模型起草的决定书中，每一项违法实体的论据和裁量细则是否完全来源于检索到的政策法条，若计算出的 Groundedness 得分低于 0.95，该文书将被标记为“存在幻觉风险”并禁止自动下发。
*   **回答相关性（Answer Relevance）**：评估生成的公文或填充的表单字段是否切实回应了最初的用户指令与表格标签语义，规避答非所问和过度赘述。

### 5.2 本地评估的两种执行策略
*   **单条追踪（Score Each Trace）**：利用 Eino Gateway 拦截器，将每一次真实执法填表的 Context、Query、Response 实时异步推送到 Ragas 进行在线打分，直接在前端 React 监控面板上进行可视化健康预警。
*   **批处理采样（Score as Batch）**：考虑到实时在线打分会占用本地显存及计算带宽，可在每日凌晨系统处于“挂起模式”时，利用本地 PostgreSQL 慢日志采样 100 条典型的文书生成会话，利用 Ragas 进行离线跑批，产出日度质量回归报告。

### 5.3 中文市监业务专属回归基准（Market Supervision DC Benchmark）
为了防止系统在调整参数（例如修改 Qdrant 检索 Top-K 限制为 6 ∼ 8，或者调整向量与 SQLite FTS5 的 RRF 混合比例）时发生检索性能衰退，系统开发团队应建立一套针对中文市场经营主体活动法律法规RAG基准数据集的回归测试，通过 500 个覆盖特种设备、广告合规、反不正当竞争等典型场景的“Query - Ground Truth Context - Standard Form”三元组，进行自动化回归测试。只有当回归指标不发生向后衰退（Regression）时，新版本代码方能通过部署，全面保障行政文档与智能化填表工作的严谨度、法律确定性与技术先进性。
